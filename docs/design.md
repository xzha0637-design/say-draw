# 设计文档 —— say-draw(AI 语音绘图工具)

> 七牛云比赛 · 命题二。本文记录:产品定位与架构思考、计划支持的指令能力、最终实现情况、未完成部分及原因。
> 题目要求的三个评审关注点 —— **指令理解的准确性与容错性、语音到绘图操作的响应延迟、复杂指令的拆解与执行能力** —— 在 §5 逐条给出设计应对与实测数据。

## 1. 题目解读与产品定位

题面第一句是产品的灵魂:**"用户不能使用鼠标或键盘,仅通过语音指令完成绘图创作。"**

我们把它读作一个真实的人,而不是一条技术限制:渐冻症 / 高位截瘫 / 帕金森患者,或双手被占用的场景。这决定了几条设计底线:

- **全流程语音可达**:不只是"画",还包括看、改、回退、收藏、管理会话 —— 任何一步需要碰键鼠,产品即不成立。
- **语音也是输出,但用户优先**:每轮操作都有 TTS 朗读回应;朗读中用户直接开口即可打断(barge-in),回声经文本相似度过滤,不会自我打断。
- **容错是底线而非加分项**:用户没有键盘可以"改错字",说错、识别错、说一半,系统都要兜住。

## 2. 路线探索史:三条被放弃的路与原因

> 本节同时是题目要求的"未完成部分的原因说明"的主体。前两条路线都有真实提交记录可查。

### 路线 A:画布图元拼装(已实现后放弃,见提交 1900661、3847f98 等)

最直观的"语音画板":语音 → 解析 → 在 Konva 画布上画圆 / 矩形 / 星星 / emoji 图元,支持编号选中、改色、移动、撤销。**做完后放弃**,原因:

- 表达力天花板太低:"猫"只能是一个圆加两个三角,或一个 emoji。"绘图创作"的"创作"无从谈起。
- 键鼠隐喻不适合语音:图形界面里拖拽 1 秒的事,语音要说三句话("选中 3 号""往左挪一点""再挪一点"),交互效率倒挂。

### 路线 B:LLM 直接绘制 SVG(已实现后放弃,见提交 172ef57)

让豆包直接输出整段 SVG 画复杂物体(房子 / 猫 / 雪人)。**能跑通但效果差**:只能画很简单的东西,稍复杂就潦草变形。根因是 **LLM 的空间想象力短板**——它擅长语义,不擅长在坐标系里维持精确的几何关系。

### 路线 C:像素板 + LLM 控制画笔(分析后否决,未实现)

设想过更激进的方案:全像素背景板,语音控制 LLM 操作画笔逐笔画,直到用户确认。分析后直接否决:

- 抽象层级比 SVG 更低,对空间精度要求更高;路线 B 已证明 LLM 在更高抽象层级上都画不好,往下走只会更糟;
- 一张图需要成百上千笔,每轮还需视觉模型回看画布纠偏,**延迟与成本是结构性的**,与评审关注的"响应延迟"正面冲突。

### 结论:抽象层级原则

```
语义描述  >  几何图元  >  SVG 路径  >  笔刷轨迹  >  像素
(LLM 越往右越弱;路线 A/B/C 分别踩在中间三档上)
```

LLM 的强项在最左端(理解、澄清、拆解语义),扩散模型的强项在最右端(把语义渲染成像素)。**正确的分工是:LLM 在语言层指挥,扩散模型在像素层执行。** 这就是最终架构。

## 3. 最终架构:语义画布(双层)+ 双路解析

### 3.1 双层:操作层与渲染层分离

```
语音(ASR, Web Speech)
  ↓
解析(双路,见 3.2)
  ↓
┌─ 操作层:语义场景图(scene)─────────────────────────┐
│  {风格, 背景, 用途, 元素[{名称,颜色,描述,方位,大小}]}      │ ← "画布"本体是这份结构化数据
│  每句话 = 对场景图的一次操作(增删元素 / 改属性 / 换背景)   │ ← 操作即时生效,~0ms
│  前端「画面要素板」实时可视化,变更卡片闪烁动画             │
└─────────────────────────────────────────────────┘
  ↓ 用户说「生成 / 可以了」才触发
┌─ 渲染层:Seedream 文生图 ──────────────────────────┐
│  场景图编译为完整 prompt → 出图(实测 ~22s)             │
│  改图:以当前版本为参考图重绘,只改所述、其余保持一致        │
└─────────────────────────────────────────────────┘
  ↓
版本检查点(每张图连同其场景图快照一起存档,可跳回任意版本分叉)
```

类比:**场景图是画面的 DOM,Seedream 是浏览器** —— 语音改 DOM,渲染器负责变成像素。昂贵的渲染由用户主动批量触发:连改五处再说一句"看看效果",只付一次出图延迟。

### 3.2 双路解析:本地快路 + 豆包慢路

- **快路(本地正则,~0ms)**:确定性高频指令——「回到第 N 张」「撤销」「收藏」「新对话」——前端直接匹配执行,零网络往返。
- **慢路(豆包,实测 1.0–1.2s)**:其余一切口语化、含糊、复杂表达,带全部对话记忆与当前场景图交给大模型,输出结构化 JSON(继续聊 / 生成 / 改图 / 多步拆解)。
- 两路协同:复杂指令被慢路拆解成步骤后,**每个步骤再分发时仍先过快路**——"回到第 2 张,把背景换成星空"中的第一步命中本地正则,0ms 执行。

## 5. 三大评审维度的设计应对

### 5.1 指令理解的准确性与容错性

- **不做关键词匹配,语义层解析**:除四类确定性指令走本地正则外,一切交给豆包按对话语义判断——"嗯…那个猫,要不弄成橙色的?"与"把猫改成橙色"等效。
- **ASR 同音字纠错**:系统提示词明确"输入来自语音识别,可能有同音错字,按发音就近纠正理解"。
- **实时字幕兜底**:临时识别结果灰色实时上屏,用户**边说边看到自己被听成了什么**,错了当场重说——没有键盘时这是第一道容错。
- **歧义反问而非乱猜**:信息不足不乱生成,AI 主动追问且尽量给出可选项("左边是猫的左边,还是画面的左边?"),开放式反问会让用户卡壳。
- **破坏性操作有退路**:任何改图都自动存版本,说错了「撤销」「回到第 N 张」即可——**可撤销比事前确认更适合语音交互**(不打断节奏)。

### 5.2 语音到绘图操作的响应延迟

延迟实测(2026-06-13,本地 Chrome + 北京区 Ark):

| 路径 | 延迟 | 说明 |
|---|---|---|
| 本地快路(回退/撤销/收藏/新对话) | **~0ms** | 前端正则,零网络往返 |
| 豆包慢路(对话/解析/拆解) | **1.0–1.2s** | `ARK_THINKING=disabled` 关思考提速 |
| Seedream 渲染(2048×2048) | **~22s** | 结构性慢,靠架构与反馈消化 |

设计应对:

- **双层架构把"操作"与"渲染"解耦**:改场景图即时生效(要素板立刻可见),22s 的渲染由用户主动批量触发——延迟不是被掩盖,而是被架构移出了高频路径。
- **先答应,再干活**:解析出生成/改图意图后,**先**朗读"好,开始画"再调渲染——感知延迟显著低于真实延迟。
- **TTS 可打断 + 话术极短**:朗读不堵嘴——用户开口即打断(回声相似度过滤防自我打断);话术仍按"三五个字"设计,确认类 ≤14 字、反问类把问题问出口。

### 5.3 复杂指令的拆解与执行能力

三个层次,各有可视化证据:

1. **一句话聊出整个画面**:"画一只戴红围巾的橙猫坐在雪山前,宫崎骏风,当头像"——慢路一次解析出主体+配饰+背景+风格+用途,**要素板上多张卡片同时浮现**(拆解结果肉眼可见)。
2. **一次改图改多处**:"猫改成白色,背景换成沙滩"——同属一次重绘的多个改动**不拆**(一次渲染更快且一致),要素板上两张卡片同时闪烁,diff 即拆解证据。
3. **跨类型复合指令才真正拆**:"回到第 2 张,再把背景换成星空"——版本操作与改图无法一次完成,豆包返回 `{action:"multi", steps:[…]}`,前端按序执行并显示步骤标签逐条打勾;每步再分发仍先过本地快路(见 3.2)。

## 6. 反馈设计:五个时刻

> 总原则:**眼睛看细节,耳朵听结论**——屏幕承载具体信息,语音只给最短确认。

| 时刻 | 视觉 | 听觉 |
|---|---|---|
| ① 说话时 | 实时字幕(灰色 interim → 定稿) | — |
| ② 听懂时 | 要素板卡片变化 / 步骤标签上屏 | 极短确认("好,改两处") |
| ③ 执行时 | 卡片闪烁动画;步骤逐条打勾 | — |
| ④ 渲染时 | 状态条"生成中…(约二十秒)" | 先播报再开始;完成后报"第 N 张好了" |
| ⑤ 出错/歧义时 | 错误信息上屏 | 教学式反问("没听明白,你可以说『把猫改成白色』") |

## 7. 与直接用豆包 APP 的区别

| 维度 | 豆包 APP(通用助手) | say-draw |
|---|---|---|
| 纯语音闭环 | 放大图 / 重生成 / 翻历史要手点屏幕 | 看、改、回退、收藏、管理会话 100% 语音可达 |
| 作品版本管理 | 对话线性,图刷过去就没了 | 检查点 + 任意跳回 + **从任一版本分叉** —— 语音版 git |
| 指令确定性 | 同一句话行为不可预期 | 高频指令本地快路 0ms;改图语义明确(参考重绘只改所述) |
| 创作状态可见 | 无 —— AI 理解了什么是黑箱 | 要素板实时呈现画面结构,缺什么、改了什么一目了然 |

一句话:豆包是"能顺便出图的聊天软件",say-draw 是**"不能动手的人也能完整完成创作、管理、迭代的工作台"**——大模型只是其中一个零件。

## 8. 未完成能力与原因

| 能力 | 未做原因 |
|---|---|
| 九宫格分镜板 | 要素板已承载"结构可视化"的核心价值;分镜板是同一思想的更重可视化,赛程内优先级让位于复合指令与文档 |
| 精确空间控制 | 扩散模型对空间关系仅粗粒度服从("右边的狗"可能跑偏),承诺像素级定位是过度承诺;只支持粗方位(左/右/上/背景)并如实声明 |
| 作品库的"语音"管理 | 多会话作品库本身已实现(登录 + `user_id×session_id` 隔离 + 服务端 SQLite 持久化 + 图片入库下载);但"打开第 N 个会话""把这幅命名为日落"等纯语音管理尚用按钮代替,语音指令集未覆盖,赛程内优先级让位于绘图主线 |
| 图片转七牛云 Kodo | 当前图片以 BLOB 入库 SQLite,单机演示足够;规模化应转对象存储(如七牛云 Kodo)只存 URL,赛程内不引入额外服务 |
| 流式 TTS(边合成边播) | Web Speech API 的合成不可流式(整句出声);换云端 TTS 可解但引入新依赖与延迟,赛程内不值。语音打断(barge-in)已实现(见能力表 #9),回声用文本相似度过滤——声学级回声消除(AEC)浏览器内不可得,极端嘈杂环境下过滤可能漏判 |
| 移动端适配 | 演示场景为桌面 Chrome;Web Speech API 在移动端浏览器表现不稳 |
