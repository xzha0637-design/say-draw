# 设计文档 —— say-draw(AI 语音绘图工具)

> 本文记录:计划支持的指令能力、最终实现情况、未完成部分及原因。随开发持续更新。

## 1. 目标与约束

- **纯语音控制**:绘图创作全程不使用鼠标/键盘(允许"按住说话"按钮)。
- **评审关注点**:指令理解的准确性与容错性、语音→操作的响应延迟、复杂指令的拆解与执行能力。

|---|---|---|---|---|

## 3. 架构

```
语音 → ASR(Web Speech) → 解析/路由(Claude Haiku,结构化输出 JSON)
     → 执行引擎(Konva 对象模型:具名图元 + 编号 + 焦点)
     → 视觉 + TTS 反馈

复杂场景(慢路):spec 分解 → 生成具名图元组(流式草图先行)
              → Opus 4.8 多模态评审纠错
```

## 4. 关键设计

- **双路架构(延迟 vs 理解力):** 简单指令走规则/快模型快路(~100ms 级);复杂指令走 LLM 慢路。两个评分维度都要照顾,不二选一。
- **选中与指代:** 每个对象带编号角标(可"选中 3 号");"它/这个"= 焦点对象(画完/选中即成焦点);歧义则 TTS 反问,绝不瞎猜。
- **草图先行:** 流式生成、边画边显,掩盖 LLM 延迟(感知延迟 ≠ 真实延迟)。
- **持续可编辑:** 生成的场景是具名图元组——几何变换=即时;语义重绘=只重画该组,其余不动。

## 5. 持久化与作品库(多画布)

> 让用户拥有多幅画、随时切换与找回——全程仍可语音操作。

- **身份与隔离:** `user_id`(用户)× `draw_id`(单幅画)双键隔离;一个用户可有多幅画,互不干扰。`user_id` 的 MVP 用浏览器本地生成的匿名 GUID(localStorage 持久),进阶再接入登录。
- **存什么:** 存结构化 spec(图元列表 / 场景 JSON)而非位图——载入后仍可继续语音编辑(呼应「持续可编辑」)。附元数据:标题、创建 / 修改时间、缩略图(可选)。
- **自动保存:** 每次操作后增量保存,免去手动「保存」的语音负担。
- **语音作品管理(示例):**「新建一幅画」「我有哪些画」「打开第二幅」「切换到上一幅」「把这幅画命名为日落」「删除这幅画」。
- **找回:** 列出该用户的全部画作(标题 + 时间 + 缩略图),语音选择载入。
- **存储选型(按投入决定,待定):**
  - 快路 / MVP:localStorage 或 IndexedDB,纯前端、零后端依赖。
  - 完整版:后端 + 持久化;本赛事可用**七牛云 Kodo 对象存储**存 spec / 缩略图(贴合主办方生态),元数据配轻量数据库。

## 6. 容错策略

- ASR 错字 / 同义词 / 模糊量词在解析层归一化;破坏性操作(清空)二次语音确认;歧义先反问再执行。
