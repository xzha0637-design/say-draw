# 全程脱手改造计划 —— 创作主回路

> **范围声明**:登录/注册允许使用键盘(用户认可)。本计划**只**保证一件事:
> 登录之后的「**出图 → 看图 → 改图 → 管理作品**」创作主回路,**全程不依赖鼠标 / 键盘**。
>
> 立项依据:题目底线「用户不能使用鼠标或键盘,仅通过语音指令完成绘图创作」(见 `docs/design.md` §1),
> 以及 README/设计文档对照表已宣称「放大图 / 翻历史 / 下载 **100% 语音可达**」——本计划负责让这句话**成立**。

---

## 0. 现状诊断:创作回路上仍有「只能点、不能说」的断点

排除登录后,主回路里以下动作**当前只有鼠标路径,没有语音路径**——对手不能动的用户,这些是死路:

| # | 创作动作 | 当前唯一入口 | 语音可达? | 代码位置 |
|---|---|---|---|---|
| G1 | 放大查看生成图(看细节再决定怎么改) | 点图 | ❌ 无 | `frontend/src/main.ts:152` |
| G2 | 关闭放大层(继续说话) | 点空白处 | ❌ 无 | `frontend/src/main.ts:140` |
| G3 | 下载 / 保存这张图 | 点 ⬇ 链接 | ❌ 无 | `frontend/src/main.ts:161` / `:124` |
| G4 | 打开「我的画」/ 切换 / 打开第 N 段会话 | 点按钮、点列表 | ❌ 无(自认于 `design.md` §8) | `frontend/src/main.ts:419` / `:343` |
| G5 | 删除会话的确认 | 原生 `confirm()`(必须点 / 敲回车) | ❌ 无 | `frontend/src/main.ts:349` |
| G6 | 开始 / 停止聆听 | 点麦克风按钮 | ⚠️ 浏览器强制首启需手势;无「停止聆听」语音 | `frontend/src/main.ts:433` |

**可语音的反而是不痛的那几个**(回退 / 撤销 / 收藏 / 新会话,见 `chat.ts` 的 `tryLocal`),
**真正卡人的看图、存图、翻作品却只能点**。本计划逐条补齐。

---

## 1. 改造原则(顺着既有架构,不重构)

- **复用本地快路**:确定性语音指令走 `Conversation.tryLocal()`(`frontend/src/chat.ts:198`),~0ms 执行,与现有「回到第N张 / 撤销 / 收藏 / 新会话」同级。新增意图只是往这条路上再挂几条正则 + 钩子。
- **能力经 `ChatUI` / `ConversationHooks` 注入**:跨模块动作(放大、下载、切会话)由 `chat.ts` 定义钩子接口,`main.ts` 落地实现——和现有 `onNewSession` 完全同形(`chat.ts:87`)。
- **每条语音指令都要有语音回执**:命中即 `ui.speak(...)`,纯语音用户才知道「听懂了、做了」。
- **干掉一切原生模态**:`confirm()` / `alert()` 本身就是键鼠依赖,一律换成应用内语音可确认的交互。

## 2. 修改方案

### P0 —— 放大 / 关闭 / 下载 语音化(改动最小,直接让对照表成立)

**① 扩展 `ChatUI` 接口**(`frontend/src/chat.ts:71`),新增三个钩子:
```ts
zoom: (n?: number) => void        // 放大当前(或第 n 张)
closeZoom: () => void             // 关闭放大层
download: (n?: number) => void    // 下载当前(或第 n 张)
```

**② 在 `tryLocal` 加意图**(`frontend/src/chat.ts:198`,放在 `if (this.versions.length === 0) return false` 之后,与 `JUMP_RE` 同级),正则定义挪到文件顶部常量区(`chat.ts:95` 附近):
```ts
const ZOOM_RE  = /^(放大|看大图|放大看看|看清楚点?|细看)\s*(?:第?\s*([0-9]+|[一二两三四五六七八九十])\s*张)?$/
const CLOSE_RE = /^(关闭|关掉|收起|退出放大|关了|看好了)$/
const DL_RE    = /^(下载|保存到本地|存到电脑|导出|下载这张)\s*(?:第?\s*([0-9]+|[一二两三四五六七八九十])\s*张)?$/
```
- 命中 `ZOOM_RE` / `DL_RE`:用现成的 `toIndex()`(`chat.ts:103`)解析可选张号,缺省取当前版本 → 调 `ui.zoom(n)` / `ui.download(n)` → `ui.speak('放大第N张')` → `return true`。
- 命中 `CLOSE_RE`:`ui.closeZoom()` → `return true`(此条不需要 `versions.length>0` 前置,放到守卫之前)。

**③ 在 `main.ts` 落地钩子**(传入 `new Conversation({...})` 的 UI 对象,`main.ts:252`):
- `zoom(n)`:取第 `n ?? 当前` 个版本的 `url` / `downloadUrl`,调现成 `openOverlay(url, downloadUrl)`(`main.ts:121`)。
  > 需让 `Conversation` 暴露「第 n 张的 url / downloadUrl」只读取数器(它已持有 `versions[]`),或直接把 zoom 解析逻辑放进 `Conversation` 再回调 `ui.zoom`。推荐后者:取数逻辑和版本数组在同一处。
- `closeZoom()`:调现成 `closeOverlay()`(`main.ts:136`)。
- `download(n)`:程序化触发下载,绕开点击——
  ```ts
  const a = document.createElement('a')
  a.href = downloadUrl
  a.setAttribute('download', '')
  a.click()
  ```
  后端 `?dl=1` 已设 `Content-Disposition: attachment`(`backend/server.js:347`),同源附件下载在 Chrome/Edge 下**无需可信用户手势**即可落到默认下载目录,真脱手。

**验收(P0)**:
- [ ] 出一张图,**只用语音**:「放大」→ 弹大图;「关了」→ 关闭;「下载」→ 文件进下载目录。
- [ ] 多张图时「放大第 2 张」「下载第 1 张」按张号命中。
- [ ] `npm run build`(含 `tsc --noEmit`)通过。

### P1 —— 作品库语音化 + 删除确认去模态

**① 念列表 + 语音开会话**。新增 `ConversationHooks`(`chat.ts:87`):
```ts
listSessions: () => Promise<{ title: string }[]>   // 供「念一遍我的画」
openSession: (n: number) => void                   // 供「打开第 N 个」
```
- 语音「**打开我的画 / 念一遍我的画**」→ 开抽屉(视觉)**并 TTS 念出列表**(「你有 3 段:1 橙猫,2 雪山…」)。纯语音用户必须**听得到**有哪些,否则抽屉对他无意义。
- 语音「**打开第 N 个 / 上一幅**」→ `openSession(n)` → 内部走现成 `switchToSession(list[n-1].id)`(`main.ts:311`)。

**② 删除确认改两段式语音**,删掉 `confirm()`(`main.ts:348-353`):
- 第一句「**删掉这段 / 删除这个会话**」→ AI 反问「确定删?说『**确认删除**』」(进入待确认态)。
- 第二句命中「确认删除」才真正 `deleteSession`;说别的则取消。逻辑放进快路,保持脱手。

**验收(P1)**:
- [ ] 只用语音:「念一遍我的画」听到列表;「打开第 2 个」切到对应会话。
- [ ] 「删掉这段」→「确认删除」两句完成删除,过程无任何原生弹窗。

# 第二部分:产品发展路线图(增长方向)

> 与上文性质不同,务必分清:
> **上文(第 0–4 节)是「债」——让现有宣称成立的必修项,先做。**
> **本部分是「增长」——产品往哪走的赌注,债还清后再谈。**
>
> 战略判断:现在做的是「一个碰巧能脱手用的语音绘图工具」。真正值得做、别人抄不走的,
> 是把视角倒过来——**从「手不能动、甚至话也说不利索」的人出发去设计的创作工具**。
> 脱手闭环 / 语义画布 / 迭代改图,只有锚在这个人身上才是护城河;否则永远在和豆包比谁出图好。
> 以下三个赌注都直接长在已有机制上。

## 赌注一:把「输入」真正占为己有(护城河 + 顺手修掉脆弱性)—— **建议优先**

**命题**:现在唯一入口是浏览器 Web Speech,它对**非标准发音**(渐冻症 / 帕金森 / 脑瘫常见的构音障碍)
识别极差——而这恰恰是目标用户。悖论:为说不清话的人做产品,却用了个只认标准普通话的识别器。

**做什么**
- [x] 接**云端流式 ASR**(七牛云 AI 流式语音识别,WebSocket)替掉 Web Speech,解掉「Chrome 依赖 Google /
      内地直连即挂 / Safari·Firefox 不支持」的脆弱性。**已实现(流式 WebSocket)**:后端 `/api/asr/stream`
      WS 代理(按七牛二进制帧协议:gzip + 帧头/序列号,key 留服务端)+ `/api/asr/status`;前端 `cloud-asr.ts`
      (Web Audio 采集 → 16k PCM 流式上送 → 累计文本按「停更 900ms」切句:增量 `onInterim`、整句 `onFinal`;
      朗读期 `setMuted` 防回环 + 能量 `onBargeIn` 打断),与 Web Speech 同接口,`main.ts` 已配置则走云端、否则自动回退。
      协议已对照七牛官方 Node/Python 示例核实。**⚠️ 待办**:配 `QINIU_AI_KEY` 后做一次端到端联调。
      「面向非典型语音 / 构音障碍」的调优为后续。
- [ ] 为完全无法稳定发声的人留一条**单开关 / 扫描式**兜底输入:界面自动轮巡高亮候选,
      用户用任意一个信号(眨眼 / 吹气 / 一个物理按键)选中。多模态无障碍。**(未做)**

**解锁**:豆包 / 通用工具**永远不会做**这个方向,也是「读懂题」的方向。做了之后,
产品定位从「语音画图」变成「**重度运动障碍者唯一能用的创作工具**」。
**复用**:对话/快路解析层不变,只换识别源;扫描式输入复用现有意图集(把「说」换成「选」)。
**依赖与风险**:需选定 ASR 供应商并就构音障碍调优(可能需要样本);扫描式是独立 UI 模式,中高工作量。
**成功判定**:构音障碍样本上识别可用率显著高于 Web Speech;无法发声者能仅凭单一信号走完一次出图。

## 赌注二:让迭代改图变快、变「局部」(把招牌从口号变成体验)

**命题**:现在每改一处都整图重抽 ~22s,既慢又贵,还可能整张漂移(见附录 A:`server.js:115` 强制 2048²)。

**做什么**
- [ ] **草图档**:先 ~2 秒出低清预览让用户确认方向(「是这意思吗」),满意再花一次全清渲染。
- [ ] **局部改(inpainting / 蒙版)**:「把左边的猫去掉」「只把围巾改蓝」只重渲染那一块,快且其余不动。

**解锁**:「语音美术指导」从「每句话等 22 秒」变成「边说边变」,配得上对照表里那句差异化。
**复用**:scene graph 已按 `name` / `pos` 存了元素,正好用来把「左边的猫」映射成蒙版区域。
**依赖与风险**:依赖图像模型的编辑/inpainting 能力与低清档;语音→区域的定位是模糊映射,需兜底反问。
**成功判定**:一次改色/去元素的端到端时延较整图重抽明显下降;改动只落在目标区域,主体不漂移。

## 赌注三:接上出图之后的「最后一公里」(产品才完整)

**命题**:现在产品止于「你有一张图」。但 scene graph 早存了 `usage`(头像/海报/表情/插画)——
知道用户要拿它干嘛,却没帮他做完。对不方便自己裁剪/排版/下单的人,**语音直达成品**才是真价值。

**做什么**
- [ ] 按 `usage` 直接产出成品:「做成微信头像」(裁圆)、「配一句字做成海报」(语音加字)、
      「做成一套表情包」、「画成四格漫画」(语音叙事天然适合分镜/连续画)。

**解锁**:从「生成器」升级为「作品工作台」,坐实 README 里「工作台」的定位。
**复用**:直接读 scene 的 `usage` 字段;分镜复用版本/会话机制做多图序列。
**依赖与风险**:多为前端后处理(裁剪/排版/打包),模型风险低,可增量上线。
**成功判定**:从「有图」到「可用成品」全程语音完成,不需任何裁图/排版的手动操作。

## 第二梯队(值得,但排在三大赌注之后)

- **个人角色 / 风格库**:「用我上次那个风格」「还是我那只柴犬」——一次定义、跨图复用
  (复用已有参考图机制做角色一致性)。
- **双向无障碍**:出图后用视觉模型**把画面讲给用户听**(低视力+运动障碍用户能「听到」自己画了什么);
  加**陪护者协同视图**,家人/护理者远程看会话、卡住时搭把手而不抢操作。
- **七牛云全家桶**:Kodo 存图 + CDN 分发(顺带修掉附录 A 的 SQLite-BLOB 扩展性问题),ASR/TTS 走七牛——
  既是基建升级,也是比赛「用对赞助商能力」的加分。

## 路线图排序

1. **先还债**:第 0–4 节脱手必修(P0 → P1 → P2)。
2. **赌注一**(护城河 + 修复输入脆弱 + 立定位)——若只推一个,推它。
3. **赌注二**(让体验配得上宣传)。
4. **赌注三**(让产品完整)。
5. 第二梯队按资源补。
